# AS5600 角度锁死 2.915 与启动卡死问题记录

## 问题现象

- 电机运行中偶发突然卡死。
- 偶发启动失败，常见于零点对齐前编码器角度在 0 到 5 度附近。
- 卡死后角度打印值长期固定在约 2.915 rad，不再随转子变化。

## 影响范围

- 编码器闭环相关模式（电流闭环、速度闭环、带弱磁速度闭环）均可能受影响。
- 问题会直接影响 Park 变换角度，导致 iq 力矩轴失效或反复错误校正。

## 复现特征

- 复现具有随机性，但在以下条件下概率更高：
  - I2C 总线存在瞬态错误。
  - 对齐阶段或启动瞬间角度接近回绕区域。
  - 系统已经发生一次通信异常但未恢复。

## 根因分析

### 根因 1：I2C 异步读缺少错误恢复闭环

旧逻辑仅依赖 MemRx 完成回调将状态从 BUSY 置为 DONE。
如果发生总线错误且没有完成回调，状态可能长期停在 BUSY 或错误状态，导致后续角度刷新链路失效。

### 根因 2：AS5600 轮询在非 DONE 状态下可能重复消费旧数据

当没有明确区分 DONE 与 ERROR/IDLE 时，轮询层可能使用到旧缓冲数据，形成“角度看起来合法但实际上未刷新”的假象。

### 根因 3：编码器链路退化后角度会固定在某一值

当原始角度长期不刷新，PLL 进入低速或停转后，`current_elec_angle` 会逼近固定相位值。
因此会看到角度长期卡在约 2.915 rad 这类常数。

## 已实施修复

### 修复 1：I2C 状态机扩展

在 `User/bsp/i2c.h` 与 `User/bsp/i2c.c` 中引入状态：
- IDLE
- BUSY
- DONE
- ERROR
- RECOVERING

并补充调试计数器：
- error_count
- timeout_count
- recover_count

### 修复 2：I2C 异常与超时恢复

在 `User/bsp/i2c.c` 中增加：
- `I2C3_ER_IRQHandler`
- `HAL_I2C_ErrorCallback`
- BUSY 超时判定（`I2C3_ASYNC_TIMEOUT_MS`）
- 错误态下 `DeInit -> Init -> Filter 配置` 的恢复流程

### 修复 3：AS5600 数据消费条件收紧

在 `User/bsp/as5600.c` 中修改轮询逻辑：
- 仅当状态为 DONE 时才消费缓冲区并返回新角度。
- 状态为 IDLE 或 ERROR 时主动重新发起异步读取。
- BUSY 或 RECOVERING 时返回未就绪，避免误用旧数据。

## 结果与现状

- 工程可正常编译、下载和运行。
- I2C 异常后可进入恢复路径，不再长期卡死在不可恢复状态。
- 角度“固定常数”现象显著降低。

## 后续验证建议

1. 连续冷启动 50 到 100 次，统计零点对齐前 0 到 5 度区间的启动成功率。
2. 运行中注入轻微干扰，观察 `i2cDebug_get_errorCount()`、`i2cDebug_get_timeoutCount()`、`i2cDebug_get_recoverCount()` 变化是否符合预期。
3. 卡死场景下同时记录：
   - 编码器原始计数是否变化
   - I2C 状态是否处于 ERROR 或 RECOVERING
   - PLL 速度是否已降为接近 0
4. 若仍有小概率失败，优先继续排查：
   - I2C 时序（1MHz 是否过激）
   - 零点对齐电压与等待时间
   - 启动瞬间电流采样一致性

## 经验总结

- 角度异常不一定是算法问题，通信链路的偶发坏样本会被控制环迅速放大。
- 对异步传感器链路，必须同时具备：状态机、超时、错误回调、恢复路径、以及严格的数据消费条件。
- 对“固定角度值”现象，优先检查上游数据是否停更，而不是先怀疑角度数学处理。
